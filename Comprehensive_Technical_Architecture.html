1. Software Architecture & Workflow

The scanner is built using Python 3, strictly following an Object-Oriented Programming (OOP) paradigm. The system consists of a centralized orchestrator (main.py) that delegates operations to a list of fully isolated scanning modules, ensuring maximum scalability and fault tolerance.

Automated Scan Lifecycle:

1
Session Initialization: Instantiates a core requests.Session object that acts as a singleton, maintaining persistency (Cookies, Headers) across all internal module requests.
2
Initial Reconnaissance: Executes standalone modules that do not require deep context (Directory Check, WAF identification, WordPress core checks, Port Sweeping).
3
Auto-Authentication: If credentials are provided via CLI, the tool crafts a dynamic request to log into the application. The authenticated session cookie is then propagated to the global state.
4
Deep Crawling: A spider traverses the application starting from the root directory, discovering hidden endpoints (up to a depth of 3 to prevent infinite loops and server overload).
5
Deep Vulnerability Analysis: Every discovered URL passes through an algorithmic filter. Forms, headers, and reflected parameters are stripped and sent to specialized modules (SQLi, XSS, LFI). Execution here is strictly Sequential to prevent SQLite database locks and session corruption on target servers.

2. Intelligent Auto-Login System

Modern applications secure endpoints with rapidly expiring cookies or heavy token dependency. Instead of manually providing a cookie, the engine was built with a smart auto-login resolver.

  • How it operates: It connects to the provided --login-url and completely parses the DOM utilizing BeautifulSoup4.
  • Field Mapping: It analyzes all <input> elements. By examining the 'name' and 'type' attributes, it mathematically infers which field requests a username (user, email) and which requests a password.
  • Hidden Token Extraction (The Complex Part): Modern frameworks (Django, Laravel, Rails) employ hidden anti-forgery tokens (e.g., csrf_token) that mutate per request. The scanner automatically extracts these dynamic inputs and merges them into the authentication POST payload.
  • Success Verification: The authentication is only marked successful if the server responds with a Protocol Redirect (301, 302) accompanied by a Set-Cookie header granting a session identifier.

3. Deep Dive into Scanner Modules

💉
SQL Injection Scanner (sqli_scan.py)
Detects 3 different variants of database injections across POST forms and GET parameters

This module is responsible for analyzing form metadata and injecting payloads. It houses the following engines:

1. Error-Based SQLi

It injects break characters such as ' or " to intentionally crash the database query. The tool utilizes an embedded dictionary containing over 20 specific error signatures across diverse database technologies (MySQL, PostgreSQL, Oracle, SQLite, and specifically Django Framework specific exceptions like django.db.utils and operationalerror).

2. Login Bypass / Logic Alteration Detection

The scanner submits a normal request alongside an injected request (e.g., ' OR '1'='1' --). If the standard request yields a 200 HTTP response (Authentication Failed), while the injected request results in a 302 Redirect (Dashboard access granted), the application immediately logs a Critical Authentication Bypass vulnerability.

3. Boolean-Blind SQLi

When database errors are suppressed by the developer, the tool uses logic inference (True/False boolean questions):

[TRUE Payload]: admin' AND 1=1 --
[FALSE Payload]: admin' AND 1=2 --

The tool calculates the byte length of the server's response. If the TRUE request length differs from the FALSE request length by more than 50 bytes, it definitively confirms blind logic injection.

📝
Cross-Site Scripting Scanner (xss_scan.py)
Frontend script injection detection and context analysis

It goes far beyond merely sending basic scripts; it analyzes where the script is reflected (Context Tracking):

  • Generic Reflection: <script>alert('XSS')</script>
  • Attribute Breakout: Attempts to break out of HTML input tags when the payload is trapped inside an attribute (e.g., an input value). Payload: "><script>alert('XSS')</script>
  • Script Context Escape: Handles inputs injected directly into embedded JavaScript maps or variables: '; alert(1); //
  • Evidence Mechanism: The system verifies if the server applied proper sanitization (e.g., converting < to &lt;). A vulnerability is confirmed only if the payload reflects verbatim as valid HTML code.
📂
Local File Inclusion (lfi_scan.py)
Probes prohibited server directories to extract high-privilege system files

A high-severity module implementing two intelligent operational methodologies:

Heuristic Parameter Extraction: The tool skips random testing. Instead, it parses HTML templates looking for contextual parameter names associated with file requests (e.g., ?file=..., ?include=, or ?path=).

WAF bypass through encoding: To bypass backend sanitization, the payloads utilize Double URL Encoding and Unicode Parsing anomalies:

Standard: ../../../../etc/passwd
Windows specific: C:\Windows\win.ini
URL Bypass: ..%2f..%2f..%2fetc%2fpasswd
Double Encoding: %252e%252e%252fetc%252fpasswd

The response text is scanned cross-referencing system signatures (e.g., root:x:0:0 for Linux or [extensions] for Windows) to definitively prove exploitation success.

📤
Unrestricted File Upload (upload_scan.py)
Simulates remote code execution (RCE) via backdoor uploads

This module identifies DOM fields comprising <input type="file"> and forcefully tests the endpoint for lack of extension whitelisting.

  • Smart Auto-fill: To avoid server rejections based on incomplete forms, the engine heuristically populates required accompanying inputs (titles, descriptions, hidden hashes) before sending the request.
  • In-Memory Attack Simulations (3 Vectors):
[1] test_scan.php (MIME: application/x-php)
Content: <?php echo 'UPLOAD_TEST'; ?>

[2] test_scan.exe (MIME: application/x-msdownload)
Content: MZ_FAKE_EXE_TEST // Mimics executable header structure

[3] test_scan.html (MIME: text/html)
Content: <script>alert(1)</script> // Mimics Stored XSS

Response codes are analyzed (200, 201) along with the presence of success indicators ("uploaded", "success") in the DOM payload. Acceptance of the execution vector files flags a High severity vulnerability.

📋
Form & CSRF Analyzer (form_scan.py)
Monitors sensitive state-changing operations for missing Cross-Site Request Forgery tokens

This unit reverse-engineers all <form> entities, extracting input names, target actions, and HTTP verbs (GET/POST).

Any POST form fundamentally alters database state (e.g., updating a profile, transferring funds). If the tool fails to identify a dynamic Anti-CSRF token input field within the form body, the system immediately registers a vulnerability based on standard framework token nomenclatures (e.g., csrfmiddlewaretoken for Django, or _token for Laravel).

🕷️
Deep Web Crawler (crawler.py)
An aggressive web spider designed specifically for reconnaissance

Unlike standard SEO spiders, this crawler maps out the entire internal attack surface area.

  • Redirect Unwrapping: It tracks 301/302 redirects, logging both the origin and the final resolution paths to ensure no endpoints are missed.
  • Regex JavaScript Parsing: Disassembles <script> tags using Regular Expressions to identify buried API endpoints mapped inside frontend routing mechanisms.
  • Static Asset Blacklisting: Automatically strips out images (JPG, PNG), stylesheets (CSS), and fonts. This optimizes the scanner speed exponentially, since static files hold no exploitable logical interactions.
📁
Directory Enumeration (dir_enum.py)
Exposes forgotten backend infrastructure files

An engine relying on a high-speed Dictionary Attack. It uses an internal list of over 50 highly lethal paths frequently abandoned by infrastructure teams.

The Wordlist is strategically categorized:

  • Admin Interfaces: /admin, /admin-panel
  • Developer Traces: /dev_logs, /backups, /console, /trace
  • Environment Variables: /.env.example, /database_schema.sql
  • Hidden APIs: /api/v1/internal/users, /api/v1/internal/transactions

It interprets 403 Forbidden responses (Path exists but is protected), 200 OK (Publicly exposed), and 301/302 redirects to definitively map the backend filesystem.

4. The Reporter & Export Ecosystem

The system utilizes a Reporter class that aggregates data from the core VulnerabilityScanner orchestrator. This engine employs a strict Deduplication Algorithm, parsing objects to ensure identical vulnerabilities (matching URL and Injection Type) are not printed twice.

The system auto-compiles 4 distinct delivery formats for security teams:

  1. HTML Documentation: A visually colored, interactive chart displaying High, Critical, and Low severities.
  2. JSON Payload: Structures data as Arrays & Objects. Built for pipeline integration or SIEM systems.
  3. CSV Matrix: Exports parameters to Excel or statistical tooling.
  4. Markdown Documentation: Syntaxes the output for seamless push to internal Github security wikis.

To adhere to global security compliance, every finding is dynamically tethered to specific mappings (OWASP Top 10 - 2021). (e.g., Injections sync to A03, Broken Access Control to A01).
